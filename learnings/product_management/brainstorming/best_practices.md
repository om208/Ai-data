## LEARN-PM-003
**Timestamp:** Thursday, 17-Apr-2026 · IST
**Keywords:** brainstorming, questions, rounds
**Domain:** product_management
**Topic:** brainstorming
**Source:** General

**Context:**
Documenting the brainstorming protocol from capability 10.

**Learning:**
Always ask 2–3 targeted questions per brainstorming round — never one
by one. Topics to cover: scope, tech preferences, feature depth. Never
ask more than 3 questions at once under any circumstance.

**Why It Matters:**
One-question-at-a-time creates a slow interrogation feel. Batching
2–3 related questions per round respects the user's time and maintains
momentum while still gathering full context.

**Apply When:**
Every brainstorming session. Before asking any question, group all
the unknowns and select the 2–3 most critical ones for this round.

## LEARN-PM-004
**Timestamp:** Thursday, 17-Apr-2026 · IST
**Keywords:** brainstorming, features, mandatory-only
**Domain:** product_management
**Topic:** brainstorming
**Source:** General

**Context:**
Documenting the feature suggestion rule from capability 10.

**Learning:**
Default to suggesting mandatory features only during brainstorming.
Suggest optional features only if the user explicitly asks for them.

**Why It Matters:**
Presenting optional features by default inflates scope perception and
overwhelms users into over-building. Mandatory-first keeps focus on
the core product that must exist before anything optional matters.

**Apply When:**
Every brainstorming session. Lead with core mandatory features.
If user asks "what else could we add?" — then introduce optional ones.

## LEARN-PM-005
**Timestamp:** Thursday, 17-Apr-2026 · IST
**Keywords:** brainstorming, research, grounded
**Domain:** product_management
**Topic:** brainstorming
**Source:** General

**Context:**
Documenting the research-grounded suggestion rule from capability 10.

**Learning:**
All feature suggestions must be grounded in research on comparable
real applications — never invented randomly. Reference specific
real-world analogues when suggesting features.

**Why It Matters:**
Random feature suggestions feel speculative and erode user trust.
Research-grounded suggestions demonstrate expertise and give the
user confidence that the recommended approach is proven.

**Apply When:**
Every time a feature or product idea is suggested. Before suggesting,
identify 1–2 comparable real applications that implement that feature.
