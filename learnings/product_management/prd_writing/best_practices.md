## LEARN-PM-001
**Timestamp:** Thursday, 17-Apr-2026 · IST
**Keywords:** prd, requirements, confirmation
**Domain:** product_management
**Topic:** prd_writing
**Source:** General

**Context:**
Initial system setup — documenting the foundational PRD writing rule from
the EA Agent capability specification.

**Learning:**
Never draft a PRD on assumptions. User confirmation of brainstormed
requirements is required before any PRD section is written.

**Why It Matters:**
PRDs written on unconfirmed assumptions create rework when requirements
change after user review — wasting effort and token budget.

**Apply When:**
Every time the `writing-prds` skill is triggered. Check that brainstorming
is confirmed and user has approved the feature list before writing.

## LEARN-PM-002
**Timestamp:** Thursday, 17-Apr-2026 · IST
**Keywords:** prd, exclusions, scope
**Domain:** product_management
**Topic:** prd_writing
**Source:** General

**Context:**
Documenting the PRD structure requirement from capability 11.

**Learning:**
A complete PRD must include an explicit exclusions section — what is
deliberately NOT being built in this version. This is as important as
the feature list.

**Why It Matters:**
Without explicit exclusions, scope creep enters through ambiguity.
Stakeholders assume anything not mentioned is included.

**Apply When:**
Every PRD creation. After listing mandatory and optional features,
always add a "What We Are NOT Building" section before edge cases.
