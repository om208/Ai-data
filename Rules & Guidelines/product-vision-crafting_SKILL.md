# Product Vision Crafting
# Version: 1.0 | Created: Tuesday, 14-Apr-2026 · IST | Updated: —

## When to use this skill
- User shares a raw idea or problem they want to solve
- Product has no defined vision or mission yet
- Existing vision needs to be sharpened or challenged
- Before writing any PRD or roadmap — vision must exist first

## Workflow
- [ ] Ask WHY — what problem exists and who has it (max 2 questions)
- [ ] Ask WHO — who is the primary user and what do they need
- [ ] Draft vision canvas using the structure below
- [ ] Show draft to user — confirm or iterate
- [ ] Finalise and save as `vision/product-vision.md`
- [ ] Re-index into Context Mode MCP immediately after save
- [ ] Verify index updated
- [ ] Extract learning → `learnings/product_management/vision/best_practices.md`
- [ ] Re-index learning file
- [ ] Hand off to `market-analyzing` or `solution-brainstorming` as next step

## Instructions

### Vision canvas structure
```md
# Product Vision — [Product Name]
# Version: 1.0 | Created: [Day, DD-Mon-YYYY · HH:MM IST]

## Problem statement
[1–2 sentences. What breaks in the world without this product.]

## Target user
[Who specifically. Not "everyone". One primary persona.]

## Value proposition
[What the product does + what makes it different from alternatives.]

## North star metric
[One number that proves the product is working.]

## Vision statement (one sentence)
"For [user] who [problem], [product] is a [category]
that [key benefit]. Unlike [alternative], [differentiator]."

## What this is NOT
[Hard boundaries. What the product will never do.]
```

### Questioning rules
- Max 2 questions per round — never interrogate
- If user is vague → offer 2–3 concrete options to choose from
- Never assume the problem — always confirm it

### Vision validation checklist
Before finalising, verify:
```
- [ ] Problem is specific — not "people waste time"
- [ ] User is specific — not "everyone" or "businesses"
- [ ] Value prop is differentiated — not identical to existing tools
- [ ] North star is measurable — not "users love it"
- [ ] NOT list is explicit — prevents scope creep later
```

→ See `resources/product-vision-crafting_resources_templates.md` for vision examples by product type.

## Quality Control Measures
- [ ] Vision confirmed by user before saving
- [ ] All 6 sections of vision canvas present
- [ ] File saved and re-indexed into Context Mode MCP
- [ ] Learning extracted and stored
- [ ] Handoff to next PM phase suggested

## Building User Trust
- Always show the vision draft before saving — never finalize silently
- Always challenge vague problem statements with specific questions
- Always suggest the logical next PM phase after vision is confirmed
